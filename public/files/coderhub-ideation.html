Feature 01 — Engineering

Show Which Test Failed & How Many Passed

Today the result is binary — pass or fail. No test count, no which case failed, no output. Step one is surfacing the pass/fail breakdown per test case. Output comparison is a stretch goal once the plumbing exists.

  • Show X / N tests passed (currently zero info on count)
  • Identify which test case failed (index + input label)
  • Stretch: show your output vs expected once data is wired
  • Sphere Engine likely returns this — we need to verify and expose it

Feature 02 — Engineering

Run Code Before Submitting

Submit is the only action today — every failed attempt costs points. A Run button lets users see output and test custom inputs before spending an attempt.

  • Synchronous run endpoint returning stdout in real time
  • Test against custom input — not just the given examples
  • Run never counts as an attempt, never deducts points
  • Execution time shown so users can optimize before submitting
Engineering — new run endpoint

def twoSum(nums, target):
  seen = {}
  for i, n in enumerate(nums):
    diff = target - n
    if diff in seen: return [seen[diff], i]
    seen[n] = i

Feature 03 — Engineering + Operations

Topics, Tracks & Learning Paths

Two connected ideas: users filter challenges by topic (Binary Search, DP…), and admins create curated Tracks — ordered collections of challenges that guide learners through a structured path.

  • Structured topic enum on challenges — replaces free-form tags
  • Filter chip bar on challenge list, single or multi-select
  • Admin creates Tracks: a named, ordered list of challenges with a goal
  • Users enroll in a Track and see progress through it
  • Ops assigns topics to existing challenges after system ships

Feature 04 — Engineering + Operations

Streak & Daily Challenge

Nothing pulls users back the next day. A streak mechanic is one of the most effective retention tools — simple to understand, hard to abandon.

  • Daily challenge rotates every 24h — curated by Ops
  • Streak counter visible on dashboard and profile
  • Streak-freeze mechanic for when life gets in the way
  • Milestone badges: 7-day, 30-day, 100-day streaks

Feature 05 — Engineering

AI Problem Assistant

The barrier to starting is often not solving the problem — it's understanding it. An in-context AI lets users ask "what does this example mean?" without leaving the page.

  • Explain the problem statement in simpler terms
  • Walk through what a given example input/output means
  • Answer "is my approach correct?" without giving the answer
  • Scoped to the current problem — not a general chatbot

Feature 06 — Engineering

Data Structure Input Types

The platform only supports flat primitives and arrays. Linked lists, trees, and graphs — the most common interview problem class — can't be expressed. This extends the input type system.

  • Extend InputTypes enum: TreeNode, ListNode, Graph
  • Visual diagram shown alongside the editor for clarity
  • Wrapper generator updated to deserialize structured inputs
  • Unlocks the most requested interview problem types

Feature 07 — Engineering (low priority)

Discussion, Editorial & Solutions Tabs

Each challenge gets three tabs: Discussion for users to share their thought process without posting solutions, Editorial for the official explanation, and Solutions for accepted code from the community.

  • Discussion tab — free-form posts, moderated, no spoilers rule
  • Editorial tab — official explanation authored by Ops per challenge
  • Solutions tab — accepted submissions shared by users (opt-in)
  • Low priority — delivering Run, output, and tracks first

Feature 08 — Engineering

Challenge Feedback Inbox for Admins

The feedback button already exists on every challenge page — users submit it, but it goes nowhere visible. Wire those submissions to an admin view so Ops can read, triage, and act on them per challenge.

  • Existing feedback button → submissions stored in DB
  • Admin inbox: list of submissions grouped by challenge
  • Each entry shows: user, message, date, current status
  • Mark as resolved / add internal note per submission
